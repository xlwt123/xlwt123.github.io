HTTP协议概述

HTTP 协议是超文本传输协议，通常承载于 TCP 协议之上，保证了数据传输的可靠性。
HTTP 协议主要规定了客户端和服务端之间的通信格式，默认使用 80 端口。
HTTP 协议是一个无状态的协议，HTTP 服务器不会保存任何客户端相关的信息，同一个客户端发送的两次请求没有任何关联关系。

HTTP/0.9

HTTP 协议最早的版本是 1991 年发布的 HTTP/0.9。该版本比较简陋，主要由客户端的 GET 请求和服务端的相应组成。

特点：

  • 只能发送 GET 请求，并且不支持请求头。
  • 服务器相应数据不包含响应头，只有一个响应体。协议规定服务器只能返回 HTML 格式的字符串，不支持返回其他格式的内容。
  • 典型的无状态性：在 TCP 连接建立后，客户端向服务端发送请求，在服务端响应后，关闭 TCP 连接。

HTTP/1.0

1996 年 5 月，HTTP/1.0 发布，支持的内容大大增加。

首先，客户端支持发送的请求方法更多了，HTTP/1.0 新增了 POST 和 HEAD 方法。
其次，HTTP 请求和响应的格式也发生了变化，每次通信都需要携带头信息，以规范请求和响应的内容和携带其他的信息。
其他的新增功能还包括状态码、支持多字符集编码、多部分发送 (multi-part type)，身份验证、缓存等。

HTTP/1.1

在 HTTP/1.0 中，虽然能力已经大大地提升了，但是仍然存在一些缺陷。在 HTTP/1.0 发布后的半年后，HTTP/1.1 发布，进一步完善了 HTTP 协议。直到现在，HTTP/1.1 都还是最流行的 HTTP 版本。

keep-alive

对比 HTTP/1.0 ，HTTP/1.1 中最大的变化，就是引入了 keep-alive 持久连接，TCP 连接默认不关闭，可以被多个请求复用。相比起一个请求结束就关闭 TCP 连接，复用 TCP 连接节省了许多时间。

客户端和服务器发现对方一段时间没有活动，就可以主动关闭连接。不过，规范的做法是，客户端在最后一个请求时，发送 Connection: close ，明确要求服务器关闭 TCP 连接。

content-length 和 分块传输

由于一个 TCP 连接中可以发送多个请求，就需要一种机制，来确保数据包是哪个请求的响应。这就是 content-length 字段的作用，声明本次响应的数据大小。

在实际的使用中，有时候需要传输大容量的数据，这样服务器必须在所有操作完成后获取到数据的大小之后才可以进行响应。这样的效率不是很高，在 HTTP/1.1 中，引入了分块传输的概念。只要请求或回应的头信息有 Transfer-Encoding 字段，就表明回应将由数量未定的数据块组成。每个非空的数据块之前，会有一个 16 进制的数值，表示这个块的长度。最后是一个大小为 0 的块，就表示本次回应的数据发送完了。

管道技术

HTTP/1.1 还引入了管道技术，在同一个 TCP 连接中，可以同时发送多个请求。不过服务器任然会按照顺序，根据请求时间的先后去做出响应。如果第一个请求十分耗时，后续的请求仍会受到影响，这就是线头阻塞。

其他功能

  • 缓存策略：

    HTTP/1.1 中，引入了更多的缓存策略：Cache-Control 和 ETag 等。

  • HOST：

    客户端请求的头信息新增了 Host 字段，用来指定服务器的域名。

  • 新增更多状态码：

    在 HTTP/1.1 中新增了 24 个错误状态响应码，如 409 (Conflict) 表示请求的资源与资源的当前状态发生冲突； 410 (Gone) 表示服务器上的某个资源被永久性的删除。

  • range:

    HTTP/1.1 中新增 range 和 content-range 字段。根据这个字段，一次请求可以只请求响应的部分内容。

HTTP/2

2015 年，HTTP/2 发布。HTTP/1.1 由于存在队头阻塞、HTTP 头部携带的信息太大和明文传输带来的不安全性等问题。为了解决这些问题，Google 推出了 SPDY 协议，也就是 HTTP/2 的前身。

二进制分帧

HTTP/2 是一个彻底的二进制协议，头信息和数据体都使用了二进制编码，统称为 “帧” (frame) 。一次请求或响应可以由 2 个或多个帧组成。

HTTP/2 将每个请求或响应的所有帧，称为一个数据流 (stream) 。每个数据流都有一个独一无二的编号，每个帧在发送的时候，都必须标记上数据流的 ID ，以区分该帧属于哪个数据流。另外还规定，客户端发出的数据流，ID 一律为奇数，服务器发出的，ID 为偶数。

数据流发送到一半的时候，客户端和服务器都可以发送信号 (RST_STREAM 帧)，取消这个数据流。HTTP/1.1 取消数据流的唯一方法，就是关闭 TCP 连接。这就是说，HTTP/2 可以取消某一次请求，同时保证 TCP 连接还打开着，可以被其他请求使用。

多路复用

由于在 HTTP/2 中，同一域名下的所有请求和响应，都在一个 TCP 连接上完成。并且由于采用了二进制分帧和流传输，请求和响应不用按照顺序一一对应，这样就解决了前代 HTTP 协议中“队头阻塞”的问题。

客户端还可以指定数据流的优先级。优先级越高，服务器就会越早回应。

首部压缩

HTTP 协议是无状态的，每次请求都必须附带上所有信息。所以，请求的很多字段都是重复的。比如 Cookie 和 User Agent ，一模一样的内容，每次请求都必须附带，这会浪费很多带宽，也影响速度。

HTTP/2 中对此作了优化，HTTP/2 对消息头采用 HPACK （专为http/2头部设计的压缩格式）进行压缩传输，能够节省消息头占用的网络的流量。同时，客户端和服务端都会维护一张头信息表，所有的字段都会存入这个表中，生成一个索引号。如果字段的内容不变，那么下次发送时只会携带索引号，节省了重复发送的开销。

服务端推送

HTTP/2 允许服务器未经请求，主动向客户端发送资源，这就是服务器推送(server push)。
